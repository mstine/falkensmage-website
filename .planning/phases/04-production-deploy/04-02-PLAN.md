**1. Create the public GitHub repo** (per D-07):

```bash
gh repo create falkensmage-website --public --source=. --remote=origin --push
```

This creates the repo under Matt's personal account (mstine), adds the `origin` remote, and pushes the current `main` branch in one command.

**2. Enable GitHub Pages with Actions source** (per research — Pitfall 2):

Immediately after repo creation, configure Pages to use GitHub Actions as the source (not "Deploy from branch"):

```bash
gh api repos/mstine/falkensmage-website/pages \
  --method POST \
  --field build_type=workflow
```

If the API call returns a 409 (already exists), update instead:
```bash
gh api repos/mstine/falkensmage-website/pages \
  --method PUT \
  --field build_type=workflow
```

**3. Set custom domain in repo Settings** (per research — authoritative for Actions deploys):

```bash
gh api repos/mstine/falkensmage-website/pages \
  --method PUT \
  --field cname=falkensmage.com
```

This is the load-bearing step for custom domain persistence. The `static/CNAME` file from Plan 01 is defense-in-depth.

**4. Verify the workflow runs**:

```bash
# Wait a moment for Actions to pick up the push
sleep 10
gh run list --limit 3
```

Check that a workflow run is in progress or completed. If the first run fails (expected if Pages wasn't enabled fast enough), re-run:

```bash
gh run rerun <run-id>
```

**5. Verify the deploy**:

Once the workflow completes successfully:
```bash
gh run view --log | tail -20
```

The site should be accessible at `https://mstine.github.io/falkensmage-website` (or at the custom domain if DNS is already pointing there). The baseURL will auto-resolve correctly once the custom domain is configured in Settings.

**Important sequencing note:** The custom domain in Settings (step 3) should be set BEFORE the first successful deploy so that `steps.pages.outputs.base_url` returns `https://falkensmage.com/` rather than the github.io subdirectory path. If the first deploy happens before the domain is set, font and CSS paths may 404 until a second deploy runs with the correct base URL.

<task type="checkpoint:human-verify" gate="blocking">
  <name>Task 2: DNS configuration + live site verification</name>
  <what-built>
GitHub repo created at github.com/mstine/falkensmage-website. GitHub Actions workflow deployed the Hugo site to GitHub Pages. Custom domain `falkensmage.com` configured in repo Settings. CNAME file present in deployed artifact.

Now Matt needs to update DNS records at Namecheap to point falkensmage.com to GitHub Pages, then verify HTTPS.
  </what-built>
  <how-to-verify>
**DNS Configuration (Namecheap):**

1. Go to Namecheap > Domain List > falkensmage.com > Manage > Advanced DNS
2. **DELETE** the existing `@` CNAME record pointing to `target.substack-custom-domains.com`
   - Important: Confirm this Substack custom domain is intentionally unused (research assumption A1). The Feral Architecture newsletter runs on feralarchitecture.com — a separate domain, unaffected by this change.
3. **ADD** four A records (Host: `@`, TTL: Automatic):
   - `185.199.108.153`
   - `185.199.109.153`
   - `185.199.110.153`
   - `185.199.111.153`
4. **ADD** one CNAME record: Host: `www`, Value: `mstine.github.io`, TTL: Automatic
5. Wait ~30 minutes for DNS propagation

**Verification commands** (run after ~30 minutes):

```bash
# Check A records resolve to GitHub Pages IPs
dig falkensmage.com A +short
# Expected: 185.199.108.153, 185.199.109.153, 185.199.110.153, 185.199.111.153

# Check www CNAME
dig www.falkensmage.com +short
# Expected: mstine.github.io

# Check HTTPS (may take 15-60 min after DNS propagates for cert provisioning)
curl -I https://falkensmage.com
# Expected: HTTP/2 200, no certificate warnings
```

**After HTTPS certificate provisions** (green checkmark in Settings > Pages):
6. Go to repo Settings > Pages > check "Enforce HTTPS" checkbox

**Final verification:**
- Open `https://falkensmage.com` in browser
- Verify the full site loads (hero image, social links, footer)
- Verify the lemniscate favicon appears in the browser tab
- Verify HTTPS padlock shows in address bar
- Push a trivial change (e.g., whitespace in hugo.toml) and verify the custom domain persists after the second deploy

**If anything is wrong:**
- If site loads but no styles/fonts: the baseURL resolved to github.io subdirectory — re-run the workflow after confirming custom domain is set in Settings
- If HTTPS shows certificate error: wait longer — certificate provisioning can take up to 60 minutes
- If custom domain resets after deploy: set it in Settings again and verify static/CNAME is in the build output

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Local -> GitHub | Git push sends all committed code to public repo |
| GitHub Actions -> GitHub Pages | Automated deployment pipeline |
| DNS -> GitHub Pages | Domain resolution from Namecheap to GitHub's CDN |
| Browser -> GitHub Pages | End users access the site over HTTPS |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-05 | Spoofing | DNS A records | mitigate | Use all four GitHub Pages A records for redundancy. Enforce HTTPS only after certificate is confirmed green in Settings. Short TTL (1800s) limits propagation window. |
| T-04-06 | Information Disclosure | Public GitHub repo | accept | Intentionally public — static personal site. No secrets, no private content, no .env files. Source code visibility is a feature. |
| T-04-07 | Denial of Service | GitHub Pages CDN | accept | GitHub Pages has built-in CDN and rate limiting. Static site with no server-side processing. No additional mitigation needed. |
| T-04-08 | Tampering | gh repo create / gh api | mitigate | All `gh` commands use authenticated GitHub CLI with Matt's token. Verify repo visibility is PUBLIC (not private) after creation. Verify Pages build_type is `workflow` (not branch). |
</threat_model>
